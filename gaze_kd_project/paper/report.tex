\begin{abstract}
We train small CNNs to regress 2D gaze $(g_x,g_y)$ from face images, with an eye toward models that could run on a phone. The implementation swaps torchvision backbones for the teacher (ResNet-18, MobileNetV2, or MobileNetV3-Small) and gives several student options, including a hand-built depthwise network (\texttt{gaze\_micro}, on the order of $50$k parameters). Besides plain MSE on labels we try knowledge distillation: add an MSE term that pulls the student toward the teacher's output. On MPIIGaze we do \emph{not} use the full official splits: we take a fixed subset of $10{,}000$ training images and $3{,}000$ validation images (deterministic ordering from the loader flags \texttt{--mpi\_max\_train\_samples} / \texttt{--mpi\_max\_val\_samples}), i.e.\ a deliberately small sample for faster runs on a course machine---not a new sample draw every epoch, just a capped pool that stays the same across epochs. ResNet-18 as a teacher still looked worse on our val subset than the smaller nets. Table numbers come from \texttt{evaluate.py} on that $3{,}000$-image validation subset.
\end{abstract}

\section{Introduction}

Gaze from a webcam or front camera shows up in HCI demos, accessibility work, and driver monitoring. The accuracy vs.\ size tradeoff is annoying: ResNets are fine on a GPU, but a 50k-parameter student is closer to what you would actually ship.

This report describes a course codebase where the task is boring on purpose: resize the crop to $224\times224$, predict two numbers, train with Adam. The only twist we cared about was KD. We also wasted a few GPU hours discovering that a ResNet-18 teacher likes to memorize our MPII train split while validation flatlines, which is why most of the plots use MobileNet-sized teachers instead.

\section{Background}

Let $\mathbf{I} \in \mathbb{R}^{3\times H \times W}$ be the input image and $\mathbf{g} = (g_x, g_y)$ the label. The network outputs $\hat{\mathbf{g}} = f_\theta(\mathbf{I})$. Supervised loss:
\[
  \mathcal{L}_{\mathrm{gt}}(\theta) = \big\lVert \hat{\mathbf{g}} - \mathbf{g} \big\rVert_2^2 .
\]
Fix a teacher $f_{\theta_T}$. Distillation adds
\[
  \mathcal{L}_{\mathrm{kd}}(\theta_S) = \big\lVert f_{\theta_S}(\mathbf{I}) - f_{\theta_T}(\mathbf{I}) \big\rVert_2^2 ,
\]
and we minimize $\mathcal{L}_{\mathrm{gt}}(\theta_S) + \alpha \mathcal{L}_{\mathrm{kd}}(\theta_S)$ with the teacher frozen. Classification KD uses temperature-soft targets; here we stayed with plain MSE on the 2-D vector because it was easy to debug.

\section{Related Work}

MPIIGaze~\cite{zhang15mpiigaze} is the dataset people cite for ``in the wild'' laptop gaze. MobileNetV3~\cite{howard19mobilenetv3} is one of the standard efficient backbones if you care about FLOPs. Hinton et al.~\cite{hinton15kd} introduced distillation for classification; we borrowed the idea but applied it to regression.

\section{Approach}

\paragraph{Data.} CSV mode expects \texttt{image\_path}, \texttt{gaze\_x}, \texttt{gaze\_y}. The loader resizes to $224\times224$ and uses ImageNet mean/std because the torchvision weights assume that.

\paragraph{Models.} Teacher: swap the classifier head for a linear layer with two outputs. Student: either another torchvision tiny model or \texttt{gaze\_micro} (depthwise separable blocks, no ImageNet init).

\paragraph{Training.} Adam, $10^{-4}$, batch $64$, usually $20$ epochs unless we say otherwise. Best checkpoint = best validation score (MSE for teacher and baseline student; combined KD loss for the distilled run). On our machine we turned on AMP (\texttt{--amp}) for speed; CPU runs skip it.

\paragraph{Metrics.} MSE, MAE, mean L2 between $\hat{\mathbf{g}}$ and $\mathbf{g}$, plus parameter count and ms/image on a dummy batch-1 forward after a short warmup (whatever \texttt{evaluate.py} prints).

\section{Experimental Results}

\subsection{Data and protocol}

We trained on MPIIGaze normalized patches via \texttt{--dataset mpiigaze}, with disjoint train vs.\ validation participants as usual (e.g.\ held-out \texttt{p14}, \texttt{p15}). For reproducibility and runtime we \textbf{subsample splits}: \texttt{--mpi\_max\_train\_samples 10000} and \texttt{--mpi\_max\_val\_samples 3000}. That means the loader exposes a fixed pool of $10{,}000$ train and $3{,}000$ val examples (deterministic order over participants/days/frames); every epoch cycles through the same pools, not a fresh random mini-corpus each time. Reported metrics are on that $3{,}000$-point validation subset only, not the full val persons.

We log curves with \texttt{--metrics\_csv}. After training, \texttt{evaluate.py --export\_json} three times, then \texttt{build\_eval\_summary.py}, then \texttt{make\_paper\_figures.py} (outputs PNG at 300\,dpi by default). The loss figure only uses the CSVs; the bar charts and size/speed figures read the merged JSON. If a KD run went 40 epochs, the plot script clips to the first 20 points by default so the x-axis is readable (\texttt{--max\_plot\_epochs}).

\subsection{Why only 10k / 3k samples}

MPIIGaze at full scale is slow to iterate on a single GPU. We capped train at $10{,}000$ and val at $3{,}000$ examples so one epoch stays short and we could actually finish ablations for the assignment. That is a self-imposed subset of the official split logic (same person-based partition, fewer frames), not a claim about matching the standard benchmark numbers. If you rerun our commands, keep both caps or the metrics in Table~\ref{tab:main} will not line up.

A proper follow-up would drop the caps (or scale them up), tune longer, and report angular error like the MPIIGaze papers do.

\subsection{ResNet-18 vs.\ smaller teachers}

We tried ResNet-18 as a teacher because it is the default ``serious'' backbone. On our MPII setup it overfit: train MSE dropped, val did not follow. MobileNetV2 / MobileNetV3-Small behaved more sanely for the same $(g_x,g_y)$ targets. So the ``big teacher'' story did not work out here; the patch-to-two-numbers mapping just is not ImageNet-classification hard.

\subsection{KD}

If the teacher is reasonable, KD sometimes shaves a bit off student val MSE relative to labels alone; sometimes it does not. $\alpha$ matters, and a bad teacher hurts. We mostly treated $\alpha$ as a knob to tune by hand.

\section{Discussion}

MSE in normalized label space is easy to implement but not what papers usually report for gaze (they want degrees). Getting angular error right needs geometry we did not implement. Augmentation (color jitter, etc.) might help but we kept the pipeline minimal so we could finish the assignment.

Exporting the student to ONNX or TFLite would be the next engineering step if this were a product. For the course, the PyTorch timings were enough.

Reproducibility: same \texttt{--mpi\_max\_train\_samples 10000}, \texttt{--mpi\_max\_val\_samples 3000}, same held-out persons, and same arch flags, or Table~\ref{tab:main} drifts.
